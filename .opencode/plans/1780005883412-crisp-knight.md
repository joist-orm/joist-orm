# GraphQL Paginated Queries Plan

## Goal

Start generating top-level GraphQL query fields, resolver scaffolds, runtime pagination helpers, and end-to-end tests, with collection query shape controlled by `paginationStyle: "cursor" | "limit"` in `joist-config.json`.

## Assumptions

- Default `paginationStyle` will be `"cursor"` so existing configs keep working without a new required field.
- Use singular entity filter names from the existing generator draft, e.g. `AuthorFilter`.
- Treat the request's `AuthorsType` / `entites` text as typos and generate `AuthorsPage { entities: [Author!]! pageInfo: PageInfo! }`.
- Do not generate `PageInfo`; the request says to assume it is an existing type.
- Generate a first-pass entity filter input per entity, e.g. `input AuthorFilter`, with primitive, enum, many-to-one, and polymorphic fields included by default.

## Implementation Steps

1. Add config support in `packages/codegen/src/config.ts`:
   - Add `paginationStyle: z.optional(z.union([z.literal("cursor"), z.literal("limit")])).default("cursor")` to the zod config object.
   - The existing exported `Config` type will pick this up automatically.

2. Pass config into schema generation:
   - Change `generateGraphqlSchemaFiles(fs, dbMeta)` to `generateGraphqlSchemaFiles(config, fs, dbMeta)`.
   - Update callers/tests accordingly.
   - Update `packages/graphql-codegen/src/index.ts` to pass `config`.

3. Generate query schema fields in `generateGraphqlSchemaFiles.ts`:
   - Re-enable `createQueryFields`, but make it accept `config`.
   - Always generate singular lookup fields:
     ```graphql
     extend type Query {
       author(id: ID!): Author!
     }
     ```
   - For `paginationStyle: "cursor"`, generate:
     ```graphql
     extend type Query {
       authors(filter: AuthorFilter, first: Int, after: String, last: Int, before: String): AuthorsConnection!
     }

     type AuthorsConnection {
       edges: [AuthorsEdge!]!
       nodes: [Author!]!
       pageInfo: PageInfo!
     }

     type AuthorsEdge {
       node: Author!
       cursor: String!
     }
     ```
   - For `paginationStyle: "limit"`, generate:
     ```graphql
     extend type Query {
       authors(filter: AuthorFilter): AuthorsPage!
     }

     type AuthorsPage {
       entities: [Author!]!
       pageInfo: PageInfo!
     }
     ```
   - Keep all generated schema entries in the existing history/upsert flow.

4. Generate entity filter inputs in `generateGraphqlSchemaFiles.ts`:
   - Add `createFilterFields(dbMeta)` to generate `input AuthorFilter` per entity.
   - Include primitive fields that can be mapped to GraphQL scalars, using nullable filter fields regardless of entity nullability.
   - Include enum and Postgres enum fields with their enum input type.
   - Include many-to-one fields as `${fieldName}Id: ID` by default, matching the save-input pattern.
   - Include polymorphic fields as `${fieldName}Id: ID` by default, matching the save-input pattern.
   - Keep the filter shape intentionally simple for the first pass: exact-match values that can be passed directly through to `em.findGql`, not nested operator inputs.
   - Include inherited base fields for STI/subclass entities using the same inherited-field pattern as entity and save-input generation.

5. Wire query resolver generation in `packages/graphql-codegen/src/index.ts`:
   - Add `generateQueryResolvers(config, dbMeta)` for singular lookups.
   - Add `generateQueryPageResolvers(config, dbMeta)` for paginated collection lookups.
   - Add `generateResolverUtils(config)` to scaffold project-level resolver utility exports.
   - Keep all generated resolver/helper scaffolds under `writeOnce` so files are only scaffolded once.

6. Fix/update `generateQueryResolvers.ts`:
   - Keep it focused on singular fields like `author`.
   - Import entity constructors via `imp(`${name}@${entitiesPath}`)` so generated snapshots include `import { Author } from "src/entities";`.
   - Use `makeRunQuery` in generated tests instead of aliasing `makeRunInputMutation`.

7. Add `generateQueryPageResolvers.ts`:
   - Generate one resolver file per entity at `resolvers/author/authorsQuery.ts`.
   - For cursor style, generate a scaffold that calls an app helper from `src/resolvers/utils`:
     ```ts
     return paginateCursor(ctx, Author, args);
     ```
   - For limit style, generate:
     ```ts
     return paginateLimit(ctx, Author, args);
     ```
   - Import `QueryResolvers`, the entity constructor, and the relevant pagination helper via `ts-poet` imports so ESM extension handling works.
   - Generate matching test scaffolds with `makeRunQuery`.

8. Add runtime pagination helpers in `packages/graphql-resolver-utils/src/pagination.ts`:
   - Export `paginateCursor`, `paginateLimit`, `CursorPageInfo`, and `LimitPageInfo` from `packages/graphql-resolver-utils/src/index.ts`.
   - Base the page info classes on the provided `JoistPageInfo` shape, but adapt to both result styles.
   - Shared inputs should work with `ctx.em`, entity constructors, GraphQL filters, and generated query args.
   - `LimitPageInfo` should use offset/limit semantics and `findCount`, with fields calculated lazily by GraphQL resolver access, not eagerly during `paginateLimit`:
     ```ts
     class LimitPageInfo<T extends Entity> {
       getHasNextPage(): Promise<boolean>;
       getTotalCount(): Promise<number>;
       getNextPage(): Promise<{ offset: number; limit: number } | undefined>;
       get currentPage(): { offset: number; limit: number };
     }
     ```
   - `CursorPageInfo` should expose standard cursor fields (`hasNextPage`, `hasPreviousPage`, `startCursor`, `endCursor`) and calculate each field lazily, so selecting only `endCursor` does not trigger count/has-next-page work.
   - Cache lazy calculations per page-info instance so selecting multiple fields that need the same count/query only runs that work once.
   - `paginateLimit` and `paginateCursor` should create page-info objects and fetch only the rows needed for the selected collection fields; they should not pre-compute page-info fields.
   - Add explicit unit tests that call only one page-info method and assert unrelated `ctx.em.findCount`/cursor work was not invoked.
   - Add integration GraphQL/resolver-style tests that request different subsets of `pageInfo` to prove lazy behavior where practical.
   - `CursorPageInfo` should use `findCount`/cursor args rather than expecting every generated resolver to duplicate pagination math.
   - Use deterministic cursors based on stable entity ids unless an existing ordering API proves more appropriate during implementation.
   - `paginateLimit(ctx, Entity, args)` should return `{ entities, pageInfo }` and call `ctx.em.findGql(Entity, args.filter ?? {}, { limit, offset })`.
   - `paginateCursor(ctx, Entity, args)` should return `{ edges, nodes, pageInfo }` and centralize cursor encoding/decoding plus page info computation.

9. Scaffold resolver utility files:
   - Add `generateResolverUtils.ts` to generate `src/resolvers/utils.ts` once with:
     ```ts
     export { entityResolver, paginateCursor, paginateLimit, saveEntity } from "joist-graphql-resolver-utils";
     ```
   - Also scaffold or update generated test utility expectations so project `src/resolvers/testUtils.ts` exports `makeRunQuery` from `joist-graphql-resolver-utils/tests` in addition to existing mutation/object helpers.
   - Existing projects may already have these files, so keep generation in the `writeOnce` flow and do not overwrite user-modified files.

10. Add codegen unit tests:
   - In `generateGraphqlSchemaFiles.test.ts`, add one default/cursor test and one explicit limit test covering query fields plus generated connection/page types.
   - Add schema snapshot coverage for generated `AuthorFilter` fields, including primitive, enum, many-to-one, and polymorphic fields.
   - Update existing snapshots to include query fields because cursor is the new default.
   - In `generateQueryResolvers.test.ts`, update snapshots for the entity import and query test helper import.
   - Add `generateQueryPageResolvers.test.ts` with cursor and limit snapshots, including import extension cases.
   - Add `generateResolverUtils.test.ts` for `src/resolvers/utils.ts` and `src/resolvers/testUtils.ts` scaffolds.

11. Add runtime/helper tests:
   - Add focused unit tests for `CursorPageInfo` and `LimitPageInfo` under `packages/graphql-resolver-utils/src`, stubbing `ctx.em.findCount`/`findGql` as needed.
   - Verify total-count behavior, next-page behavior, no-next-page behavior, and cursor/page info fields.

12. Add end-to-end tests:
   - Set `"paginationStyle": "limit"` in `packages/tests/integration/joist-config.json`.
   - Update `packages/tests/integration/src/resolvers/utils.ts` to export `paginateLimit`/`paginateCursor` from `joist-graphql-resolver-utils`.
   - Update `packages/tests/integration/src/resolvers/testUtils.ts` to export `makeRunQuery`.
   - Add an integration resolver test similar to `packages/tests/esm/src/resolvers/author/saveAuthorMutation.test.ts`, e.g. `packages/tests/integration/src/resolvers/author/authorsQuery.test.ts`, that creates several authors and verifies the generated `authors` limit pagination resolver returns entities plus page info.
   - In the integration test suite, include separate filter cases proving `paginateLimit` pipes filters to `em.findGql` correctly for primitive, enum, many-to-one, and polymorphic fields.
   - Use existing integration entities where possible for filter coverage; if a polymorphic author-facing filter is not available, choose the smallest existing polymorphic entity/resolver or add the minimal schema fixture needed.
   - Add a new minimal `packages/tests/cursor` project with an `authors` table, `paginationStyle: "cursor"`, GraphQL codegen wiring, generated resolver utils, and a resolver test that verifies cursor pagination works over multiple authors.
   - In the cursor test suite, include primitive, enum, many-to-one, and polymorphic filter cases proving `paginateCursor` pipes filters to `em.findGql` correctly.
