Joist 2.1.0 is the first stable release under our [dual release channels](./dual-release-channels) approach, bundling up ~two months of work from the `next` channel into an official release.

Here's what's new.

## Breaking Change: PostgresDriver Now Accepts `pg.Pool`

The biggest change in 2.1.0 is that `PostgresDriver` now accepts a `node-pg` `Pool` directly, rather than a Knex instance ([#1754](https://github.com/joist-orm/joist-orm/pull/1754)).

Previously, you'd initialize Joist with:

```typescript
import Knex from "knex";
import { PostgresDriver } from "joist-orm";

const knex = Knex({ client: "pg", connection: connectionConfig });
const driver = new PostgresDriver(knex);
```

In 2.1.0, you pass a `pg.Pool` directly:

```typescript
import { Pool } from "pg";
import { PostgresDriver } from "joist-orm";

const pool = new Pool(connectionConfig);
const driver = new PostgresDriver(pool);
```

Removing Knex from the driver reduces Joist's dependency footprint and gives us full control over how we interact with Postgres. All batch operations (insert, update, delete, m2m) now use `pg.PoolClient` directly, with `?` placeholder-style SQL converted to Postgres-native `$1, $2, ...` format internally. Transaction management uses explicit `BEGIN`/`COMMIT`/`ROLLBACK` on the `PoolClient`.

If you're using `joist-knex` for schema management helpers, those are unchanged — Knex is still available there. Only the runtime driver initialization changes.

## Recursive Many-to-Many Collections

Joist now supports self-referential many-to-many relationships ([#1739](https://github.com/joist-orm/joist-orm/pull/1739), [#1744](https://github.com/joist-orm/joist-orm/pull/1744)).

The classic example is a tagging system where tags can have related tags:

```sql
CREATE TABLE tags (id SERIAL PRIMARY KEY, name TEXT NOT NULL);
CREATE TABLE tag_to_tags (tag_id INT REFERENCES tags, related_tag_id INT REFERENCES tags);
```

Joist's codegen will now generate `children` and `parents` collections on the `Tag` entity (using the existing `hasRecursiveChildren`/`hasRecursiveParents` naming convention), so you can traverse the graph:

```typescript
const tag = await em.load(Tag, "t:1", { children: { children: {} } });
// Walk the tree
for (const child of tag.children.get) {
  console.log(child.children.get);
}
```

These collections also work inside reactive hints, so reactive fields and rules that span the recursive relationship will automatically recompute when the graph changes.

## EntityManager.pendingChanges

A new `em.pendingChanges` getter provides a structured view of all work buffered in the `EntityManager` before the next `flush()` ([#1748](https://github.com/joist-orm/joist-orm/pull/1748)):

```typescript
const changes = em.pendingChanges;
// [
//   { kind: "create", entity: Author#1 },
//   { kind: "update", entity: Book#2 },
//   { kind: "delete", entity: Publisher#3 },
//   { kind: "m2m", op: "add", joinTableName: "book_to_tags", entities: [Book#2, Tag#5] },
// ]
```

This is equivalent to scanning `em.entities` and checking `pendingOperation` yourself, but provides a more declarative API — useful for audit logging, pre-flush validation, or instrumentation that needs to inspect what's about to be written.

## Factory Cross-Referencing with `is`

Test factories now support a `is` alias for cross-referencing entities within a single factory call ([#1762](https://github.com/joist-orm/joist-orm/pull/1762)).

When building a subgraph where one entity needs to reference a sibling being created in the same call, you previously had to split the construction across multiple statements:

```typescript
const [b1] = newAuthor(em, { books: [{ title: "First Book" }] }).books.get;
const a = newAuthor(em, {
  books: [b1, { title: "Second Book", prequel: b1 }],
});
```

With `is`, you can express this in one call:

```typescript
const a = newAuthor(em, {
  books: [
    { is: "b#1", title: "First Book" },
    { is: "b#2", title: "Second Book", prequel: "b#1" },
  ],
});
const [b1, b2] = a.books.get;
expect(b2.prequel.get).toEqual(b1);
```

The `is` key asserts the entity gets the expected factory id (e.g. `b#1` means "the 1st `Book` created in this `EntityManager`"). If the assertion fails, an error is thrown — which helps catch test rot as factories evolve.

The `"b#1"` string reference syntax also works _without_ `is` — `is` just adds the assertion. And if you prefer not to destructure, the `factories` proxy gives direct access:

```typescript
import { factories } from "joist-orm";

newAuthor(em, {
  books: [
    { is: "b#1", title: "First Book" },
    { is: "b#2", title: "Second Book", prequel: "b#1" },
  ],
});

const { b1, b2 } = factories;
expect(b2.prequel.get).toEqual(b1);
```

## Allow `OneToManyCollection.set` When Unloaded

Previously, calling `.set()` on a `OneToManyCollection` that hadn't been loaded yet would throw. This was an unnecessary restriction — Joist now queues the set and applies it correctly during flush ([#1725](https://github.com/joist-orm/joist-orm/pull/1725)).

This comes up in code that initializes or replaces a collection without caring about its prior contents:

```typescript
// No longer need to ensure author.books is loaded before setting
author.books.set([book1, book2]);
await em.flush();
```

## More Readable SQL Aliases

Joist's join-preloading plugin generates SQL aliases like `a0`, `b1`, `b2` for joined tables. These are now generated using the entity's table name abbreviation rather than a purely sequential counter ([#1731](https://github.com/joist-orm/joist-orm/pull/1731)).

So instead of a preload query with joins aliased `a0`, `b1`, `b2`, `c3`, you'll see `a0`, `b0`, `b1`, `p0` — giving you a hint of what entity each join corresponds to when reading `EXPLAIN` output or query logs.

## Performance: Fewer Promises

A round of refactoring reduced the number of `Promise` allocations in Joist's hot paths ([#1730](https://github.com/joist-orm/joist-orm/pull/1730)). This is part of ongoing performance work following the [pipelining benchmarks](./pipelining-alpha-benchmark) and the [unnest work](./bringing-back-unnest). No API changes — the gains should be transparent.

## Codegen Syncs Markdown and JS Docs

`joist-codegen` now reads JSDoc comments and markdown documentation from your `joist.config.json` and propagates them into the generated entity codegen files ([#1749](https://github.com/joist-orm/joist-orm/pull/1749)).

This means you can document fields once in your config:

```json
{
  "entities": {
    "Author": {
      "fields": {
        "firstName": {
          "comment": "The author's legal first name."
        }
      }
    }
  }
}
```

And the generated codegen file will include that comment on the getter, so it's visible in IDE tooltips.

## Bug Fixes

- **Fix stale preload cache causing dropped entities** ([#1741](https://github.com/joist-orm/joist-orm/pull/1741)): A critical fix. When `em.recalc` (or any `populate` with `forceReload: true`) ran on an entity whose o2m collection had unflushed children, those children could be silently dropped from the collection after the next `em.flush()`. The root cause was that `#preloadedRelations` wasn't cleared during flush, so a stale cache entry from before the flush could override the collection on a subsequent forced reload.

- **Fix m2o missing reactions** ([#1750](https://github.com/joist-orm/joist-orm/pull/1750)): Reactive fields and rules that depended on a many-to-one relation could fail to recompute in certain cases.

- **Rename `addCycleMessage` to `addCycleRule`** ([#1747](https://github.com/joist-orm/joist-orm/pull/1747)): The method name better reflects that it's adding a validation rule, not just a message. The old name is removed.

- **Fix `addCycleRule` support for o2m recursive collections** ([#1745](https://github.com/joist-orm/joist-orm/pull/1745)): Cycle detection wasn't working correctly for one-to-many recursive collections (e.g. `parent`/`children` trees).

- **Fix reloading recursive collections** ([#1735](https://github.com/joist-orm/joist-orm/pull/1735)): Reloading a recursive collection after mutations could produce incorrect results in some cases.

- **Remove implicit `LIMIT` and `DISTINCT` from `findIds`** ([#1758](https://github.com/joist-orm/joist-orm/pull/1758)): These clauses were being added unnecessarily and could cause incorrect results when a caller passed their own `LIMIT`.

- **Do not use `em.entityLimit` for m2m rows** ([#1773](https://github.com/joist-orm/joist-orm/pull/1773)): The entity limit check (a safety guard against runaway loads) was incorrectly applied to join table rows, which aren't entities.

- **Don't accidentally drop the migrations table** ([#1760](https://github.com/joist-orm/joist-orm/pull/1760)): A bug in `flush_database` helper logic could cause the migrations table to be dropped. Fixed.

- **Add more entity limit checks** ([#1775](https://github.com/joist-orm/joist-orm/pull/1775)): Improves safety for applications running with a configured `entityLimit`.

- **ReactiveReference refactored to state machine** ([#1774](https://github.com/joist-orm/joist-orm/pull/1774)): Internal refactor continuing the state machine work done for o2m and m2m collections in recent releases. Improves correctness and maintainability of reactive reference lifecycle tracking.

## Upgrading

The only breaking change is the `PostgresDriver` initialization — swap your `Knex` instance for a `pg.Pool`. Everything else is backwards compatible.

As always, the `@next` channel will continue to receive changes as we merge them, and the next stable release will follow in a couple of months.
