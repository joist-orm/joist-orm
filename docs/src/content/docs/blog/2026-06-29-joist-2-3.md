Joist [2.3](https://github.com/joist-orm/joist-orm/commit/108b79314224bca1a75ef5bc6fc8071f8f6c18b6) is out! This release has a mix of headline features, query improvements, and a big round of performance work.

The biggest changes are:

- Rails-style `scope`s for naming and composing reusable `em.find` filters
- Scaffolding for GraphQL query resolvers
- Paginated `em.find` calls are now auto-batched
- New `EnumCollection`s for many-to-many join tables that point at an enum
- Native `Temporal` support on Node 26, with no polyfill required
- Smarter batched queries via constant inlining and boolean pruning
- A large round of hot-path performance work, with 20-50%+ speedups across flush, hydration, and `find`

## Rails-Style Scopes

The headline 2.3 feature is [#1892](https://github.com/joist-orm/joist-orm/pull/1892): Rails-style scopes 🎉, which let you name and compose common `em.find` filters directly on your entity classes.

You declare scopes as static fields, using the `scope` function:

```ts
export class Author extends AuthorCodegen {
  static adult = scope({ age: { gte: 18 } });
  static active = scope({ deletedAt: null });
  static popular = scope((a) => a.isPopular.eq(true));
  static hasBooks = scope({ books: true });
  static booksReviewedBy = scope.fn((reviewer: Author) => ({ books: { reviewer } }));
}
```

And then reuse and chain them throughout your codebase:

```ts
await Author.adult.find(em);
await Author.adult.popular.find(em);
await Author.active.where({ firstName: "a1" }).find(em);
await Author.booksReviewedBy(reviewer).find(em);
```

Scopes are syntax-sugar for Joist's regular [em.find](/features/queries-find), so they share the same filter syntax and semantics, including relation filters across one-to-many, many-to-one, and many-to-many paths.

A few things scopes can do:

- **Chain with `AND` semantics**, i.e. `Author.adult.popular` finds authors that are both, and you can define a scope in terms of another: `static popularAdult = Author.popular.adult`.
- **Take parameters** via `scope.fn`, i.e. `Author.named("a")` or `Author.taggedWith("fiction")`.
- **Add ad-hoc builders**, i.e. `Author.adult.orderBy({ createdAt: "DESC" }).limit(10)` or `.softDeletes("include")`.
- **Drop into `em.find` anywhere a filter is expected**, i.e. `em.find(Author, Author.adult)` or `em.find(Book, { author: Author.adult })`.

Because they compile down to regular `em.find` queries, scopes also support join-alias conditions for predicates a plain nested filter can't express, like a top-level `or` that spans two different joined tables:

```ts
export class Author extends AuthorCodegen {
  // Authors with a book that is either titled "b1" or has a 3-star review.
  static titleOrRated = scope((a) => {
    const [b, r] = aliases(Book, BookReview);
    return {
      where: { books: { as: b, reviews: { as: r } } },
      conditions: { or: [b.title.eq("b1"), r.rating.eq(3)] },
    };
  });
}
```

Joist's scopes are heavily inspired by Rails scopes 🙏, but are strongly-typed and adapted to fit Joist's conventions--most notably, scopes always require a trailing `.find(em)` (or `findOne`, `findOneOrFail`, `findCount`, `findIds`) so Joist knows which `EntityManager` to use for loading and identity caching.

:::caution

To achieve the chained/recursive `Author.adult.active` typing, Joist generates an `AuthorScopes` interface in each `<Entity>Codegen.ts`, so you should re-run `joist-codegen` after each change to your `scope` declarations.

:::

See the new [Scope Queries](/features/scope-queries) docs for the full feature set.

## GraphQL Query Scaffolding

Joist has had great support for scaffolding GraphQL object resolvers & save mutations for awhile now, but we just finally got around to adding query resolver scaffolding in [#1865](https://github.com/joist-orm/joist-orm/pull/1865).

Now an `Author` entity will get scaffolded single and list queries automatically created:

```graphql
extend type Query {
  author(id: ID!): Author
  authors(filter: AuthorFilter, limit: Int, offset: Int): AuthorsPage!
}
```

As well as default implementations:

```ts
export const authors: Pick<QueryResolvers, "authors"> = {
  async authors(_, args, ctx) {
    return paginate(ctx, Author, args);
  },
};
```

And, as always, if you need to change any of the scaffolded output, just update it, and Joist will never overwrite your changes.

## Batched Paginated Finds

Joist's `em.find` has always auto-batched semantically-similar queries in flight at the same time, but historically that did not work for _paginated_ finds, which is a common N+1 trap for GraphQL fields like `reviews(first: 5)` nested deep in a query.

In 2.3, [#1836](https://github.com/joist-orm/joist-orm/pull/1836) batches paginated `em.find` calls by moving the `LIMIT` into a `CROSS JOIN LATERAL` subquery, so each parent gets its own per-row `ORDER BY ... LIMIT`, while Postgres still only sees one query.

Because that removes the only reason for a separate, deliberately-unbatched method, [#1838](https://github.com/joist-orm/joist-orm/pull/1838) also removes `em.findPaginated` entirely. Now any `em.find` can accept `limit`/`offset` and still get Joist's auto-batching:

```ts
const [aReviews, bReviews] = await Promise.all([
  em.find(BookReview, { book: b1 }, { limit: 5, orderBy: { id: "ASC" } }),
  em.find(BookReview, { book: b2 }, { limit: 5, orderBy: { id: "ASC" } }),
]);
```

We covered this in detail in its own [Batched Paginated Finds](/blog/batched-paginated-finds) post.

:::caution

This is a breaking change: `em.findPaginated` is gone--replace its calls with `em.find` using the same `limit`/`offset` options.

:::

## Enum Collections

Joist has always supported "lists of enum values" via `int[]` enum-array columns, but [#1908](https://github.com/joist-orm/joist-orm/pull/1908) adds a second option: a true many-to-many join table between an entity and an enum, modeled as an `EnumCollection`.

For example, to give `Publisher` a list of "logo colors", create a join table between `publishers` and the `color` enum table:

```ts
createManyToManyTable(b, "publisher_logo_colors", "publishers", { table: "color", column: "logo_color_id" });
```

Joist recognizes that one side points at an enum table and generates an `EnumCollection` on the entity side (only the entity side; the enum does not get a reverse relation):

```ts
readonly logoColors: EnumCollection<Publisher, Color> = hasEnumCollection();
```

To the caller it looks like "an array of `Color`s", but unlike an enum-array column it's **lazy** (it lives in its own table, so it needs a load hint), and it's fully integrated with the rest of Joist:

```ts
const publisher = await em.load(Publisher, "p:1", "logoColors");
publisher.logoColors.get; // [Color.Red, Color.Blue]
publisher.logoColors.add(Color.Green);
publisher.logoColors.set([Color.Green, Color.Blue]);
await publisher.logoColors.includes(Color.Blue); // probe without loading
```

It supports filtering (`em.find(Publisher, { logoColors: Color.Red })`), `changes` tracking, and reactivity, just like any other collection. See the [Enum Collections](/modeling/enum-tables/#enum-collections) docs for when to reach for this versus a plain enum-array column.

## Native Temporal on Node 26

Joist [supports the `Temporal` API](/getting-started/configuration/#temporal) for date/time columns, and historically required the `temporal-polyfill` package to provide it.

Node 26 ships `Temporal` natively, so [#1889](https://github.com/joist-orm/joist-orm/pull/1889) updates Joist's native-first / polyfill-fallback detection to transparently use the global `Temporal` when it's available, falling back to the polyfill on older Node versions. There's also a new `import { Temporal } from "joist-orm"` that resolves to whichever implementation is in use, so codegen and app code don't need a direct import to either.

## Smarter Batched Queries

Two changes make Joist's batched `em.find` queries leaner for the Postgres planner.

[#1909](https://github.com/joist-orm/joist-orm/pull/1909) adds **constant inlining**. Previously, every filter value in a batch flowed through the `_find` CTE, even columns whose value was identical across all the batched calls. Now Joist detects when all batched queries share the same value for a column and inlines it as a plain bound predicate, leaving only the genuinely-varying columns in the CTE:

```sql
WITH _find (tag, arg0) AS (
  SELECT unnest($1::int[]), unnest($2::character varying[])
)
SELECT array_agg(_find.tag) as _tags, a.*
FROM authors AS a
CROSS JOIN _find AS _find
WHERE a.deleted_at IS NULL AND a.first_name = $3 AND a.last_name = _find.arg0
GROUP BY a.id
```

Here `first_name = $3` is a plain scalar comparison the planner can satisfy with an index, while only `last_name`--the column that actually differs between the batched queries--flows through the CTE.

[#1879](https://github.com/joist-orm/joist-orm/pull/1879) adds **boolean expression pruning**, so a common conditional-filter idiom just works:

```ts
em.find(Author, {
  name: { ilike: name && `${name}%` },
});
```

When `name` is falsy, the `ilike: false` term is pruned from the query entirely (rather than generating a spurious predicate), while equality operators like `eq: false` are preserved.

## Many-to-Many Tables Without `id` Columns

[#1904](https://github.com/joist-orm/joist-orm/pull/1904) lets Joist work with many-to-many join tables that don't have a surrogate `id` primary key, i.e. tables whose primary key is just the `(foo_id, bar_id)` pair. This is a common shape for join tables, and Joist now handles loading, preloading, and mutating their rows just like the `id`-backed variant.

## Performance Work

2.3 includes a large round of hot-path performance work, benchmarked at 100k-entity scale across several independent process runs. The headline numbers (vs. the prior commit, all confirmed):

- **Flush scanning** ([#1857](https://github.com/joist-orm/joist-orm/pull/1857)): with a mostly-clean unit of work, scanning for dirty entities went from `~4ms` to `~0.1ms` for the 0-dirty case (`~97%`), and `~9.5ms` to `~4.8ms` with 100 dirty entities (`~49%`).
- **Entity writer binding** ([#1857](https://github.com/joist-orm/joist-orm/pull/1857)): insert-heavy flushes dropped `~28-47%` depending on column count.
- **Lazy find indexes** ([#1861](https://github.com/joist-orm/joist-orm/pull/1861)): building a find index lazily, only when fields are actually queried, took a first-build scenario from `~1517ms` to `~24ms` (`~98%`), with `30-50%` steady-state wins.
- **Find filter cache keys** ([#1859](https://github.com/joist-orm/joist-orm/pull/1859)): query prep dropped `~52-58%`.
- **Reactive queue lookups** ([#1857](https://github.com/joist-orm/joist-orm/pull/1857)): `~22-43%` faster.
- **Entity hydration** and **identity-map `loadAll`** paths: `~17-34%` faster.
- **Already-loaded `populate`** paths: `~22-33%` faster.

There's also a new `BENCHMARK-PLAN.md` with the methodology and a regression-benchmark suite to keep these from regressing.

## Smaller But Notable Changes

A few other changes worth calling out:

- [#1900](https://github.com/joist-orm/joist-orm/pull/1900) aligns many-to-one and polymorphic behavior on empty arrays.
- [#1883](https://github.com/joist-orm/joist-orm/pull/1883) excludes `em.deleted` entities from `findCount`, and [#1871](https://github.com/joist-orm/joist-orm/pull/1871) ensures `find` does not return pending-delete entities.
- [#1881](https://github.com/joist-orm/joist-orm/pull/1881) always defines `transientFields`.
- [#1874](https://github.com/joist-orm/joist-orm/pull/1874) caches loader promises within an entity, and [#1868](https://github.com/joist-orm/joist-orm/pull/1868) avoids recursive-property populate deadlocks.
- [#1873](https://github.com/joist-orm/joist-orm/pull/1873) handles cycles in recursive Properties.
- [#1891](https://github.com/joist-orm/joist-orm/pull/1891) skips preloading GraphQL fields with arguments.
- [#1863](https://github.com/joist-orm/joist-orm/pull/1863) restores `em.flush` return order, and [#1862](https://github.com/joist-orm/joist-orm/pull/1862) keeps `loadLens` synchronous for already-loaded paths.

There are also many correctness fixes around STI subtypes, recursive relations, reaction handling, and dependency bumps, plus codegen conflict-avoidance fixes ([#1901](https://github.com/joist-orm/joist-orm/pull/1901), [#1902](https://github.com/joist-orm/joist-orm/pull/1902), [#1903](https://github.com/joist-orm/joist-orm/pull/1903)) to keep generated scope/enum/type names from colliding with user symbols.

## Upgrading

The main migration notes are:

- Replace `em.findPaginated` calls with `em.find`, passing the same `limit`/`offset` options.
- After adding any `scope` declarations, re-run `joist-codegen` to refresh the generated `<Entity>Scopes` types.

Thanks to everyone filing issues, testing `next` releases, and pushing on Joist's edge cases.
