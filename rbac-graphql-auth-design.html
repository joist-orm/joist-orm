Our Webapp RBAC Design Doc

Dynamic Permissions in a GraphQL UI

Our webapp currently has:

  • static GraphQL queries
  • static navigation
  • static form layouts

But our proposed RBAC-based auth will be dynamic per user, per page, per operation, and sometimes per field. I.e. a user could be able to edit this author name, but not that author name.

This document compares three API shapes for making the UI adapt without wrapping every field, button, table column, and call-to-action in handwritten permission checks.

The Core Problem

Backend authorization enforces reads and writes. The frontend still needs enough permission metadata to make the experience honest: hide fields the user cannot see, disable fields they cannot edit, disable saves they cannot perform, and avoid GraphQL requests that fail because a static screen queried a denied field.

Questions the UI Must Answer

For any given entity, such as Author id=1:

  • Can this user view author.name?
  • Can this user edit author.email?
  • Can this user see a private author.royalty field?
  • Can this user create a new Book for this author?
  • Can this user click edit, save, or archive?

Mental Model

The UI needs two permission categories:

  • Field permissions decide whether a value is visible and editable.
  • Operation permissions decide whether a command is available, such as create, edit, save, promote, archive, or delete.

View

If denied, the value should not render and should not accidentally leak through form state, cache, or network payloads.

Edit

If read-only, the value may render but input components should be disabled and save payloads should avoid sending changes.

Operate

Page-level actions such as create, edit, save, promote, and archive need their own permission signals.

Approach 1: Low-Level Permission Calls

The frontend first asks for raw permission names for an entity or page. That blocking response is then used to dynamically construct the author data query so denied fields are not requested. UI components directly check permission strings such as Author.name.edit.

GraphQL API Calls

query AuthorPermissions($id: ID!) {
  permissions(entityId: $id) {
    name
    allowed
  }
}

query AuthorEditPage($id: ID!) {
  author(id: $id) {
    id
    name
    email
    books { id title }
  }
}

Example Response

{
  "entityId": "a:1",
  "permissions": [
    { "name": "Author.edit", "allowed": true },
    { "name": "Book.create", "allowed": false },
    { "name": "Author.name.view", "allowed": true },
    { "name": "Author.name.edit", "allowed": true },
    { "name": "Author.email.view", "allowed": true },
    { "name": "Author.email.edit", "allowed": false },
    { "name": "Author.royaltyRate.view", "allowed": false }
  ]
}

Frontend Component Shape

function AuthorEditPage(props) {
  const permissions = useAuthorPermissionsQuery({ id: props.id });
  const can = createPermissionLookup(permissions.data?.permissions ?? []);
  const author = useAuthorQuery({
    id: props.id,
    fields: can("Author.royaltyRate.view") ? ["name", "email", "royaltyRate"] : ["name", "email"],
  });

  return (
    <AuthorForm
      author={author.data?.author}
      canEditAuthor={can("Author.edit")}
      canEditName={can("Author.name.edit")}
      canViewRoyaltyRate={can("Author.royaltyRate.view")}
    />
  );
}

function AuthorForm(props) {
  return (
    <Form>
      <TextField name="name" disabled={!props.canEditName} />
      {props.canViewRoyaltyRate ? <RoyaltyRateField /> : null}
      <SaveButton disabled={!props.canEditAuthor} />
    </Form>
  );
}

Pros

  • Maximum frontend flexibility.
  • Raw permission names are visible for debugging and advanced UI decisions.
  • Can be introduced without reshaping every existing entity field.

Cons

  • Frontend becomes coupled to low-level backend permission names.
  • Requires separate data and permission queries that must be merged.
  • Does not solve denied static field queries by itself.
  • Permission checks can spread across many components.

Approach 2: Graph-Embedded Auth

The GraphQL schema exposes decorated field objects and operation objects. Instead of querying name, the page queries nameField { value canView canEdit }. Reusable form components can understand a standard field permission fragment.

GraphQL API Call

fragment EditableStringField on EditableStringField {
  value
  canView
  canEdit
  deniedReason
}

fragment OperationPermission on OperationPermission {
  allowed
  deniedReason
}

query AuthorEditPage($id: ID!) {
  author(id: $id) {
    id
    nameField { ...EditableStringField }
    emailField { ...EditableStringField }
    royaltyRateField { ...EditableStringField }
    books { id titleField { ...EditableStringField } }
    canEdit { ...OperationPermission }
    canSave { ...OperationPermission }
  }
}

Example Response

{
  "author": {
    "id": "a:1",
    "nameField": {
      "value": "Octavia Butler",
      "canView": true,
      "canEdit": true,
      "deniedReason": null
    },
    "emailField": {
      "value": "octavia@example.com",
      "canView": true,
      "canEdit": false,
      "deniedReason": "Role cannot edit author email"
    },
    "royaltyRateField": {
      "value": null,
      "canView": false,
      "canEdit": false,
      "deniedReason": "Royalty fields are restricted"
    },
    "canEdit": { "allowed": true, "deniedReason": null },
    "canSave": { "allowed": true, "deniedReason": null }
  }
}

Frontend Component Shape

function AuthorEditPage(props) {
  const result = useAuthorEditPageQuery({ id: props.id });
  const author = result.data?.author;

  return author ? <AuthorForm author={author} /> : null;
}

function AuthorForm(props) {
  return (
    <Form>
      <EditableTextField label="Name" field={props.author.nameField} name="name" />
      <EditableTextField label="Email" field={props.author.emailField} name="email" />
      <EditableTextField label="Royalty rate" field={props.author.royaltyRateField} name="royaltyRate" />
      <SaveButton disabled={!props.author.canSave.allowed} />
    </Form>
  );
}

function EditableTextField(props) {
  if (!props.field.canView) return null;
  return <TextField name={props.name} value={props.field.value ?? ""} disabled={!props.field.canEdit} />;
}

Pros

  • No separate permission bookkeeping query.
  • Backend can hide low-level permission names behind semantic booleans.
  • Standard fragments enable reusable components like EditableTextField.
  • Denied values can safely resolve to null plus metadata instead of throwing.

Cons

  • Every screen and query must migrate from raw fields to decorated fields.
  • Form state maps from nameField.value to mutation input name, adding boilerplate.
  • If the UI needs raw policy details, the API must expose new semantic flags.
  • Schema surface area grows substantially.

Approach 3: Tagged Values

The client opts in to tagged scalar values. Existing fields can still be queried, but string values return with permission prefixes such as rw:Octavia Butler, r:octavia@example.com, or d:none. Beam components learn how to interpret tags.

GraphQL API Call

query AuthorEditPage($id: ID!) {
  author(id: $id) {
    id
    name
    email
    royaltyRate
  }
}
Header: X-Tagged-Permission-Values: true

Example Response

{
  "author": {
    "id": "a:1",
    "name": "rw:Octavia Butler",
    "email": "r:octavia@example.com",
    "royaltyRate": "d:none"
  }
}

Frontend Component Shape

function AuthorForm(props) {
  return (
    <Form>
      <BeamTextField name="name" value={props.author.name} />
      <BeamTextField name="email" value={props.author.email} />
      <BeamTextField name="royaltyRate" value={props.author.royaltyRate} />
    </Form>
  );
}

function BeamTextField(props) {
  const tagged = parsePermissionTaggedString(props.value);
  if (tagged.access === "denied") return null;

  return (
    <TextField
      name={props.name}
      value={tagged.value}
      disabled={tagged.access === "read"}
    />
  );
}

Pros

  • Least disruption to existing GraphQL query shapes.
  • Can reduce the risk of old screens failing on denied fields.
  • Centralized Beam components can interpret tags once.

Cons

  • Only works naturally for strings.
  • Numbers, dates, booleans, collections, and references need separate mechanisms.
  • Pollutes scalar values with transport metadata.
  • Easy to forget to strip or reapply tags in form state, validation, and mutations.
  • Does not model operation permissions like create, save, edit, or promote.

Comparison

Dimension Low-Level Permission Calls Graph-Embedded Auth Tagged Values
API shape Separate data query and permission query. Entity fields include value plus permission metadata. Existing scalar fields return tagged values after opt-in.
Frontend coupling High coupling to raw permission names. Coupled to semantic field and operation objects. Coupled to tag protocol in every scalar-consuming path.
Migration cost Medium. Add permission query and merge logic per page. High. Rewrite queries and form bindings from raw fields to decorated fields. Low for strings, high once non-string types are included.
Safety for denied fields Incomplete unless the data query also avoids denied fields. Strong. Denied field can return null value with explicit canView false. Mixed. Avoids hard failure, but metadata travels as data.
Component reuse Requires helper lookups or prop plumbing. Strong fit for reusable EditableField components. Strong only for Beam components that consume supported scalar types.
Operation permissions Good fit as raw permission strings. Good fit as canSave, canEdit, canCreate. Not addressed by scalar tags.

Recommended Direction

The clean long-term model is Approach 2: Graph-Embedded Auth, optionally paired with a narrow compatibility bridge inspired by Approach 1 for page-level operations and migration diagnostics. Tagged values are attractive as a migration safety valve, but they appear too fragile as the primary model because our webapp fields are not only strings.

1. Standardize Field Wrappers

Introduce reusable GraphQL field types such as EditableStringField, EditableMoneyField, and EditableDateField.

2. Standardize Operations

Expose page/entity operations as semantic objects: canCreate, canEdit, canSave, canPromote.

3. Hide Boilerplate in Components

Build Beam equivalents of bound fields that accept field metadata and handle hide, disable, help text, and denied reasons.
